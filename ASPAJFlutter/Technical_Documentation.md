# 🛠️ Technical Documentation: 4LLASET System

## Architecture Overview
The system follows a decoupled architecture:
- **Backend**: Laravel 11.x (PHP 8.2+) providing a RESTful API and an Admin Dashboard.
- **Frontend**: Flutter 3.2.3+ focusing on a role-based mobile/web experience.
- **Real-time**: Pusher (configured via environment variables).
- **Email**: SMTP compatible (Testing with Mailpit).
- **Bug Reporting**: Dedicated API endpoint for submitting diagnostic information and screenshots.

## Database Schema (Normalization)
The database `db_asetkejuruan` has been optimized:
1.  **Users**: Multi-role system (admin, officer, student).
2.  **Students**: Linked to `User` and `SchoolClass`.
3.  **Commodities**: Assets linked to specific `jurusan` (departments).
4.  **Borrowings**: 
    - `borrowings`: Parent record for a transaction.
    - `borrowing_items`: Child records allowing multiple assets per transaction.
    - **Inventory Safeguard**: Stock is deducted on request and restored on rejection.

## Key Accessors (PHP)
- `Commodity::getPhotoUrlAttribute()`: Handles varied image path formats and provides a default asset image if missing.
- `Borrowing::getFullBorrowDatetimeAttribute()`: Combines date and time fields for display.

## API Integration (Flutter)
The `ApiService` uses a static `BASE_URL`. For production, this must be updated to the server's static IP or domain name.

## Troubleshooting CORS
If the Flutter web app fails to fetch data, ensure the `cors.php` config in Laravel allows the production domain and headers:
```php
'paths' => ['api/*', 'sanctum/csrf-cookie', 'login', 'logout', 'register', 'cart/*'],
'allowed_methods' => ['*'],
'allowed_origins' => ['*'],
```
